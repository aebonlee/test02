## 📁 전체 디렉토리 구조 (한눈에 보기)

```
프로젝트_루트/
│
# ========== 진행 프로세스 (P0~S5) ==========
├── Process/                      # 모든 단계 폴더
│   │
│   # === 특별단계 (P0, S0) - 프로젝트 기초 ===
│   ├── P0_작업_디렉토리_구조_생성/    # 프로젝트 구조 및 상태 관리 ⭐ 핵심 문서
│   │   ├── Project_Directory_Structure.md   # 이 파일!
│   │   └── Project_Status.md                # 프로젝트 진행 상황
│   ├── S0_Project-SAL-Grid_생성/          # 프로젝트 SAL Grid 관리 시스템
│   │
│   # === 예비단계 (P1-P3) - GRID 범위 밖 ===
│   ├── P1_사업계획/                  # 비즈니스 계획
│   ├── P2_프로젝트_기획/              # 프로젝트 기획
│   ├── P3_프로토타입_제작/            # 프로토타입 제작
│   │
│   # === 실행단계 (S1-S5) - GRID 관리 ===
│   ├── S1_개발_준비/                 # 개발 환경 준비
│   ├── S2_개발-1차/                  # 1차 개발 (핵심 기능)
│   ├── S3_개발-2차/                  # 2차 개발 (확장 기능)
│   ├── S4_개발-3차/                  # 3차 개발 (고급 기능)
│   └── S5_개발_마무리/               # 개발 마무리 (배포/안정화)
│
# ========== 배포용 코드 (루트 폴더) ==========
├── api/                         # 백엔드 API
├── pages/                       # 프론트엔드 페이지
├── assets/                      # 정적 자원 (CSS, JS, 이미지)
│
# ========== 설정 폴더 ==========
├── .claude/                     # Claude Code 설정
├── .git/                        # Git 저장소
├── scripts/                     # 자동화 스크립트
│
# ========== 루트 파일 ==========
├── .gitignore                   # Git 제외 파일 목록
└── README.md                    # 프로젝트 설명
```

## 🎯 디렉토리 네이밍 규칙

### 규칙 1: 특별단계는 P0, S0
```
✅ Process/P0_작업_디렉토리_구조_생성/    # 프로젝트 구조 문서
✅ Process/S0_Project-SAL-Grid_생성/          # 프로젝트 SAL Grid
```

### 규칙 2: 예비단계는 P + 숫자
```
✅ Process/P1_사업계획/
✅ Process/P2_프로젝트_기획/
✅ Process/P3_프로토타입_제작/
```

### 규칙 3: 실행단계는 S + 숫자
```
✅ Process/S1_개발_준비/
✅ Process/S2_개발-1차/
✅ Process/S3_개발-2차/
✅ Process/S4_개발-3차/
✅ Process/S5_개발_마무리/
```

### 규칙 4: 하위 폴더는 영문 사용
```
✅ Process/P2_프로젝트_기획/User_Flows/
✅ Process/S2_개발-1차/Backend_APIs/
❌ Process/S2_개발-1차/백엔드_API/  (한글 사용 X)
```

### 규칙 5: 파일명 규칙

**파일명: kebab-case 사용 (소문자 + 하이픈)**
```
✅ 좋은 예:
- google-login.js
- user-profile.js
- email-send.js

❌ 나쁜 예:
- googleLogin.js        (camelCase 금지)
- google_login.js       (snake_case 금지)
```

**파일 상단: Task ID 주석 필수**
```javascript
/**
 * @task S2BA1
 */
export default async function handler(req, res) {
  // ...
}
```

## 📂 Stage 폴더 공통 구조

각 Stage 폴더 (S1~S5)는 11개의 Area 폴더를 포함할 수 있습니다:

```
S{N}_단계명/
├── Backend_APIs/         # BA: 백엔드 API
├── Backend_Infra/        # BI: 백엔드 인프라
├── Content_System/       # C: 콘텐츠 시스템
├── Database/             # D: 데이터베이스
├── Design/               # U: 디자인
├── DevOps/               # O: 운영/배포
├── Documentation/        # M: 문서화
├── External/             # E: 외부 연동
├── Frontend/             # F: 프론트엔드
├── Security/             # S: 보안/인증
└── Testing/              # T: 테스트
```

## 📦 배포 구조 (루트 폴더)

```
루트/                           ← 배포 루트 디렉토리
├── api/                       ← 백엔드 API (Area별 분류)
│   ├── Backend_APIs/          # BA Area
│   ├── Security/              # S Area
│   ├── Backend_Infra/         # BI Area
│   └── External/              # E Area
│
├── pages/                     ← 화면 (F Area)
│   └── auth/                  # 인증 관련
│
├── assets/                    ← 정적 자원
│   ├── css/                   # 스타일시트
│   ├── js/                    # JavaScript
│   └── images/                # 이미지
│
├── scripts/                   ← 자동화 스크립트 (개발용)
├── index.html                 ← 메인 페이지
└── 404.html                   ← 에러 페이지
```

**Stage → 루트 복사 규칙:**
| Stage 폴더 | 루트 위치 |
|------------|----------|
| `S?_*/Frontend/` | `pages/` |
| `S?_*/Backend_APIs/` | `api/Backend_APIs/` |
| `S?_*/Security/` | `api/Security/` |
| `S?_*/Backend_Infra/` | `api/Backend_Infra/` |
| `S?_*/External/` | `api/External/` |

## 💡 참고: React/Next.js로 전환할 경우

> 현재 이 문서는 **Vanilla JavaScript 프로젝트**의 구조를 설명합니다.
> React/Next.js 프로젝트로 전환하면 폴더 구조가 달라집니다.

### Vanilla 방식 (현재)

```
루트/
├── api/                    ← 백엔드 API
├── pages/                  ← 화면/페이지
├── assets/                 ← 정적 자원
├── scripts/                ← 자동화 스크립트
├── index.html              ← 메인 페이지
└── 404.html                ← 에러 페이지
```

**특징:** 6개 요소 (4개 폴더 + 2개 HTML 파일)

### React/Next.js 방식 (전환 시)

```
루트/
├── app/                    ← 페이지 + API + 레이아웃 통합
├── components/             ← 재사용 UI 조각
├── public/                 ← 정적 자원 (Vanilla의 assets)
└── lib/                    ← 유틸리티, 클라이언트 설정
```

**특징:** 4개 폴더만 필요 (HTML 파일 불필요, 더 간결함)

### 폴더 매핑 (Vanilla → React)

| Vanilla | React/Next.js |
|---------|---------------|
| `pages/` | `app/` |
| `api/` | `app/api/` |
| `assets/` | `public/` |
| `index.html` | `app/page.tsx` |
| `404.html` | `app/not-found.tsx` |

### 언제 전환해야 할까?

- **Vanilla로 시작 권장**: 프로토타입 단계에서는 Vanilla가 더 빠르고 간단
- **React로 전환 고려**: 프로토타입 완료 후, 복잡한 UI 상호작용이 필요할 때
- **전환 시점 판단**: P3 프로토타입 제작 단계 완료 후 결정

**상세 내용:** `부수적_고유기능/콘텐츠/실전_Tips/개발_실무/Vanilla와_React란_무엇인가.md` 참조

## 🛑 절대 규칙

### 새 폴더 생성 전 사용자 승인 필수

```
⛔ 새 폴더를 만들기 전에 반드시 사용자에게 물어야 합니다!
⛔ 임의로 폴더를 생성하면 프로젝트 구조가 엉망이 됩니다!
```

**새 폴더 생성이 필요할 때:**
1. 기존 폴더 중 적합한 곳이 있는지 먼저 확인
2. 정말 없다면 작업 중단
3. 사용자에게 질문: "폴더 [폴더명] 생성이 필요합니다. 승인하시겠습니까?"
4. 승인 후에만 폴더 생성
