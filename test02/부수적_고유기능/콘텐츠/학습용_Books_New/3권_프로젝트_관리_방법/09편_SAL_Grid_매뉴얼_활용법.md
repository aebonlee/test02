# 9편 | SAL Grid 매뉴얼 활용법

---

새 가전제품을 사면 두꺼운 사용 설명서가 딸려온다. 대부분의 사람들은 이 설명서를 읽지 않는다. 일단 전원을 켜고 버튼을 눌러본다. 문제가 생기면 그때 설명서를 찾는다.

SAL Grid 매뉴얼도 비슷하다. 170KB 분량에 27개 섹션이나 되는 방대한 문서다. 처음부터 끝까지 읽으려고 하면 지친다. 그리고 다 읽어도 실제로 필요할 때 어디에 뭐가 있었는지 기억나지 않는다.

이 편에서는 매뉴얼을 효과적으로 활용하는 방법을 알아본다. 전체를 다 읽지 않고도 필요한 정보를 빠르게 찾는 방법이다.

## 1. 매뉴얼의 역할

매뉴얼은 SAL Grid의 모든 것을 담은 종합 레퍼런스 문서다. 몇 가지 역할을 한다.

레퍼런스 역할을 한다. "Task ID는 어떻게 만들더라?" 같은 질문에 정확한 답을 제공한다. 기억이 안 날 때 찾아보면 된다.

가이드 역할을 한다. "Stage Gate 검증은 어떻게 하지?" 같은 절차를 안내한다. 단계별로 따라하면 된다.

트러블슈팅 역할을 한다. "Supabase 연결이 안 돼" 같은 문제의 해결책을 제공한다. 문제 상황과 해결 방법이 정리되어 있다.

표준 역할을 한다. 팀원들이 같은 규칙을 따르도록 기준을 제공한다. "나는 이렇게 하는데 너는 왜 다르게 해?"라는 논쟁이 줄어든다.

## 2. 7개 파트 구조

매뉴얼은 7개 파트로 나뉜다. 각 파트는 관련된 주제들을 묶어놓았다.

파트 1은 개요와 프레임워크다. SAL Grid가 무엇인지, 5×11 Matrix가 무엇인지, 22개 속성이 무엇인지 설명한다. 처음 시작할 때 한 번 읽어두면 좋다.

파트 2는 Grid 생성이다. Task를 어떻게 나누는지, Instruction을 어떻게 쓰는지 다룬다. 새 프로젝트를 시작하거나 Task를 추가할 때 참조한다.

파트 3은 검증과 추적 시스템이다. 3단계 검증, Stage Gate, Git 연동 등을 다룬다. 검증 단계에서 참조한다.

파트 4는 Supabase 설정과 실행이다. 데이터베이스 설치부터 CRUD 작업까지 다룬다. Supabase 관련 작업을 할 때 참조한다.

파트 5는 Viewer와 자동화다. HTML Viewer 사용법과 Claude Code 자동화를 다룬다. Viewer를 설정하거나 자동화 스크립트를 만들 때 참조한다.

파트 6은 문제 해결과 운영이다. 자주 발생하는 문제와 해결책, 보안 설정, 백업 방법을 다룬다. 문제가 생겼을 때 여기를 먼저 찾아본다.

파트 7은 정리와 부록이다. 핵심 내용 요약과 버전 이력이 있다. 빠르게 복습하거나 변경 내역을 확인할 때 본다.

## 3. 필요한 정보 찾기

"Task를 어떻게 나눠야 하지?" 파트 2의 섹션 6으로 간다. Task 선정 원칙이 상세히 설명되어 있다.

"Instruction은 어떻게 쓰지?" 파트 2의 섹션 7과 8로 간다. Task Instruction 작성법과 Verification Instruction 작성법이 있다.

"검증은 어떻게 하지?" 파트 3의 섹션 9와 10으로 간다. 3단계 검증 시스템과 Stage Gate 시스템이 설명되어 있다.

"Supabase가 연결이 안 돼" 파트 6의 섹션 22로 간다. 문제 해결 A to Z에 흔한 오류와 해결책이 정리되어 있다.

"22개 속성이 뭐였더라?" 파트 1의 섹션 3으로 간다. 22개 속성 정의가 하나씩 설명되어 있다.

매뉴얼 파일에서 Ctrl+F(또는 Cmd+F)를 눌러 키워드로 검색할 수도 있다. "dependencies"를 검색하면 의존성 관련 내용이 나오고, "Stage Gate"를 검색하면 Gate 관련 내용이 나온다.

## 4. MD 버전과 HTML 버전

매뉴얼은 두 가지 형태로 존재한다. 원본인 MD 파일과 변환된 HTML 파일이다.

MD 버전은 편집용이다. 매뉴얼을 수정할 때 이 파일을 편집한다. 마크다운 문법으로 작성되어 있어서 텍스트 에디터에서 바로 수정할 수 있다. Git으로 버전 관리도 된다.

HTML 버전은 조회용이다. 브라우저에서 바로 열어볼 수 있다. 목차가 있어서 클릭하면 해당 섹션으로 이동한다. 마크다운을 모르는 비개발자도 쉽게 읽을 수 있다.

AI가 참조할 때는 MD 버전을 읽는다. 파일에서 직접 텍스트를 추출하기 편하기 때문이다.

사람이 빠르게 조회할 때는 HTML 버전이 편하다. 스크롤하면서 찾는 것보다 목차 클릭이 빠르다.

## 5. AI의 매뉴얼 활용

AI도 매뉴얼을 참조한다. 하지만 170KB 전체를 매번 읽지는 않는다.

새 세션이 시작되면 먼저 CLAUDE.md를 읽는다. 여기에 핵심 규칙이 요약되어 있다. 대부분의 작업은 이 요약만으로 충분하다.

상세한 내용이 필요하면 rules 폴더의 규칙 파일을 참조한다. 파일 명명 규칙, 저장 위치 규칙, Area/Stage 규칙 등이 별도 파일로 분리되어 있다. 짧고 구체적이라 찾기 쉽다.

규칙 파일에도 없는 내용이면 그때 매뉴얼을 찾는다. Stage Gate 리포트 형식이라든지, Supabase 테이블 스키마 같은 상세 정보.

이 순서를 따르면 효율적이다. 매번 170KB를 읽는 것보다 필요한 부분만 찾는 것이 빠르다.

## 6. 매뉴얼 업데이트

매뉴얼은 살아있는 문서다. 규칙이 바뀌거나, 새로운 기능이 추가되거나, 오류가 발견되면 업데이트한다.

업데이트할 때는 해당 섹션을 수정하고, 버전 번호를 올린다. v3.4에서 v3.5로. 그리고 섹션 27 버전 이력에 변경 내용을 기록한다. 언제 무엇이 바뀌었는지 추적할 수 있다.

MD 파일을 수정한 후에는 HTML 파일도 업데이트해야 한다. 빌드 스크립트를 실행하면 MD가 HTML로 변환된다. 이 과정을 빠뜨리면 MD와 HTML이 서로 다른 내용을 보여주게 된다.

## 7. 활용 팁

몇 가지 팁을 알아두면 매뉴얼을 더 효과적으로 활용할 수 있다.

목차를 먼저 확인하자. 27개 섹션이 있으니 목차를 보고 원하는 섹션으로 바로 이동하는 것이 좋다. 처음부터 쭉 읽는 것보다 훨씬 빠르다.

파트 단위로 파악하자. 전체를 다 읽기보다 현재 작업에 해당하는 파트만 집중해서 읽는다. Grid를 만들고 있으면 파트 2만, 검증 중이면 파트 3만, 문제가 생겼으면 파트 6만.

예시를 중심으로 이해하자. 정의와 설명을 읽는 것보다 예시를 보는 것이 더 빨리 이해된다. 매뉴얼에는 많은 예시가 포함되어 있다.

버전을 확인하자. 매뉴얼 상단에 버전 번호와 최종 수정일이 있다. 오래된 버전을 보고 있으면 최신 규칙이 반영되지 않았을 수 있다.
