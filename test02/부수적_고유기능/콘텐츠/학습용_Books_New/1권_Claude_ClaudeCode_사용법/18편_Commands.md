# 18편 | Commands

---

Claude Code는 슬래시(/)로 시작하는 명령어를 제공한다. `/help`를 입력하면 도움말이 나오고, `/clear`를 입력하면 대화가 초기화된다. 이 편에서는 Claude Code가 제공하는 명령어와 직접 명령어를 만드는 방법을 살펴본다.

## 1. Commands 개요

### 1-1 Commands란

Commands는 슬래시(/)로 시작하는 특수 명령어이다. 일반 대화와 달리 특정 기능을 즉시 실행한다.

```
일반 대화: "설정을 열어줘"
명령어:    /config
```

명령어는 두 종류가 있다:
- **제공 명령어**: Claude Code가 기본으로 제공 (약 40개)
- **사용자 정의 명령어**: 직접 만들어서 사용

### 1-2 명령어 확인 방법

```
/help
```

`/help`를 입력하면 사용 가능한 모든 명령어 목록이 표시된다. 제공 명령어와 사용자 정의 명령어가 함께 나온다.

## 2. 세션 관리

대화 세션을 관리하는 명령어이다.

| 명령어 | 기능 |
|--------|------|
| `/clear` | 대화 내역 초기화 |
| `/exit` | Claude Code 종료 |
| `/rename <이름>` | 세션 이름 변경 |
| `/resume` | 이전 대화 재개 |
| `/export` | 대화 내보내기 |

### /clear

대화 내역을 전부 삭제하고 새로 시작한다.

```
/clear
→ 대화가 초기화되었습니다.
```

**주의:** 되돌릴 수 없다.

### /resume

이전에 진행했던 대화를 이어서 한다.

```
/resume
→ 세션 목록 표시
→ 원하는 세션 선택
→ 이전 대화 이어서 진행
```

특정 세션을 바로 열 수도 있다:
```
/resume "프로젝트 A 작업"
```

### /export

현재 대화를 파일로 저장하거나 클립보드에 복사한다.

```
/export                    → 클립보드에 복사
/export "conversation.md"  → 파일로 저장
```

## 3. 코드 작업

코드 관련 작업을 수행하는 명령어이다.

| 명령어 | 기능 |
|--------|------|
| `/review` | 코드 리뷰 요청 |
| `/security-review` | 보안 검토 |
| `/rewind` | 변경사항 되돌리기 |
| `/pr-comments` | PR 댓글 확인 |

### /review

현재 코드에 대한 리뷰를 요청한다.

```
/review
→ 코드 품질, 보안, 성능 등을 검토
→ 개선 제안 제공
```

### /rewind

Claude Code가 수정한 코드를 이전 상태로 되돌린다.

```
/rewind
→ 변경 내역 목록 표시
→ 되돌릴 시점 선택
→ 코드 복원
```

작업 중 실수가 발생했을 때 유용하다.

### /security-review

보안 관점에서 코드를 검토한다.

```
/security-review
→ 현재 브랜치의 변경사항 분석
→ 보안 취약점 확인
→ 개선 방안 제시
```

## 4. 설정 및 상태

설정을 변경하거나 상태를 확인하는 명령어이다.

| 명령어 | 기능 |
|--------|------|
| `/config` | 설정 화면 열기 |
| `/status` | 상태 확인 |
| `/model` | 모델 변경 |
| `/permissions` | 권한 관리 |
| `/output-style` | 출력 스타일 변경 |

### /config

설정 화면을 연다.

```
/config
→ 설정 UI 표시
→ 다양한 옵션 변경 가능
```

### /model

사용할 AI 모델을 변경한다.

```
/model
→ 모델 목록 표시
→ claude-opus, claude-sonnet, claude-haiku 등 선택
```

직접 지정도 가능하다:
```
/model claude-sonnet
```

### /output-style

Claude의 응답 스타일을 변경한다.

```
/output-style
→ 스타일 옵션 표시
→ 간결함, 상세함 등 선택
```

## 5. 효율성 관리

토큰 사용과 비용을 관리하는 명령어이다.

| 명령어 | 기능 |
|--------|------|
| `/context` | 컨텍스트 사용량 확인 |
| `/cost` | 토큰 비용 확인 |
| `/compact` | 대화 압축 |
| `/usage` | 사용량 한도 확인 |
| `/todos` | TODO 목록 확인 |

### /context

현재 컨텍스트 사용량을 시각적으로 보여준다.

```
/context
→ 토큰 사용량 그래프
→ 남은 용량 확인
```

대화가 길어지면 컨텍스트 한계에 가까워지는데, 이때 확인하면 좋다.

### /cost

현재 세션의 토큰 사용량과 비용을 확인한다.

```
/cost
→ 입력 토큰: 15,000
→ 출력 토큰: 8,000
→ 예상 비용: $0.25
```

### /compact

대화 내용을 압축하여 토큰을 절약한다.

```
/compact
→ 대화 내용 요약
→ 토큰 사용량 감소
```

압축 방향을 지정할 수도 있다:
```
/compact "에러 해결 과정만 유지해줘"
```

### /todos

현재 대화에서 식별된 TODO 항목을 보여준다.

```
/todos
→ 대기 중인 작업 목록 표시
```

## 6. 환경 설정

프로젝트 환경을 설정하는 명령어이다.

| 명령어 | 기능 |
|--------|------|
| `/init` | 프로젝트 초기화 |
| `/memory` | 메모리 파일 편집 |
| `/add-dir` | 작업 디렉토리 추가 |
| `/terminal-setup` | 터미널 설정 |
| `/vim` | Vim 모드 |

### /init

프로젝트에 CLAUDE.md 파일을 생성한다.

```
/init
→ CLAUDE.md 파일 생성
→ 프로젝트 가이드 작성
```

CLAUDE.md는 Claude Code에게 프로젝트에 대한 정보를 알려주는 파일이다.

### /memory

CLAUDE.md 파일을 직접 편집한다.

```
/memory
→ CLAUDE.md 편집 모드
→ 프로젝트 정보 추가/수정
```

### /add-dir

작업 범위에 추가 디렉토리를 포함한다.

```
/add-dir
→ 디렉토리 선택
→ 작업 범위 확장
```

여러 프로젝트를 동시에 다룰 때 유용하다.

### /vim

Vim 편집 모드로 전환한다.

```
/vim
→ Vim 모드 활성화
→ insert/command 모드 전환 가능
```

Vim에 익숙한 사용자를 위한 기능이다.

## 7. 확장 및 연동

외부 도구와 연동하는 명령어이다.

| 명령어 | 기능 |
|--------|------|
| `/mcp` | MCP 서버 관리 |
| `/plugin` | 플러그인 관리 |
| `/hooks` | Hook 설정 |
| `/install-github-app` | GitHub 연동 |
| `/ide` | IDE 통합 |

### /mcp

MCP(Model Context Protocol) 서버를 관리한다.

```
/mcp
→ 연결된 MCP 서버 목록
→ 사용 가능한 도구 확인
```

MCP는 Claude Code의 기능을 확장하는 프로토콜이다 (23편에서 상세 설명).

### /plugin

플러그인을 설치하고 관리한다.

```
/plugin
→ 플러그인 관리 메뉴

/plugin install <이름>
→ 특정 플러그인 설치
```

플러그인은 Commands, Skills, Hooks 등을 패키지로 묶은 것이다 (23편에서 상세 설명).

### /hooks

이벤트에 반응하는 Hook을 설정한다.

```
/hooks
→ Hook 설정 관리
→ 도구 실행 전후 동작 정의
```

Hooks는 21편에서 상세히 설명한다.

### /install-github-app

GitHub Actions와 연동을 설정한다.

```
/install-github-app
→ GitHub 리포지토리 연결
→ PR에서 @claude 멘션 가능
```

### /ide

IDE 통합 상태를 확인하고 관리한다.

```
/ide
→ VS Code, JetBrains 등 연동 상태
→ 통합 설정 관리
```

## 9. 사용자 정의 명령어

직접 명령어를 만들어서 사용할 수 있다.

### 9-1 만드는 방법

**1단계: 폴더 생성**

프로젝트 루트에 `.claude/commands/` 폴더를 만든다.

```
프로젝트/
└── .claude/
    └── commands/
        └── 명령어이름.md
```

**2단계: 명령어 파일 작성**

마크다운 파일로 명령어 내용을 작성한다.

```markdown
# commit.md

현재 변경사항을 커밋합니다.

## 실행할 작업
1. git status 확인
2. 변경사항 스테이징
3. Conventional Commits 형식으로 커밋 메시지 작성
4. 커밋 실행
```

**3단계: 사용**

```
/commit
→ 명령어 파일 내용대로 실행
```

### 9-2 매개변수 사용

명령어에 인자를 전달할 수 있다.

**$ARGUMENTS - 모든 인자:**
```markdown
# deploy.md
$ARGUMENTS 환경에 배포합니다.
```

```
/deploy production
→ "production 환경에 배포합니다."로 실행
```

**$1, $2, $3 - 개별 인자:**
```markdown
# create-component.md
$1 이름으로 $2 타입의 컴포넌트를 생성합니다.
```

```
/create-component Button functional
→ "Button 이름으로 functional 타입의 컴포넌트를 생성합니다."
```

### 9-3 활용 예시

**커밋 자동화 (/commit):**
```markdown
현재 변경사항을 커밋하고 푸시합니다.

1. git status 확인
2. 변경사항 스테이징
3. Conventional Commits 형식으로 커밋
4. 원격에 푸시
```

**테스트 실행 (/test):**
```markdown
테스트를 실행하고 결과를 보고합니다.

1. 단위 테스트 실행
2. 통합 테스트 실행
3. 커버리지 확인
4. 실패한 테스트 상세 분석
```

**배포 체크 (/deploy-check):**
```markdown
배포 전 체크리스트를 확인합니다.

## 확인 항목
- [ ] 모든 테스트 통과
- [ ] 빌드 성공
- [ ] 보안 취약점 없음
- [ ] 환경 변수 설정 완료
```

**코드 리뷰 (/review):**
```markdown
코드를 리뷰하고 피드백을 제공합니다.

## 검토 항목
1. 코드 품질
2. 보안 취약점
3. 성능 문제
4. 모범 사례 준수
```

**콘텐츠 평가 (/evaluate):**
```markdown
콘텐츠 품질을 100점 만점으로 평가합니다.

## 평가 기준 (각 20점)
- 기술적 정확성
- 가독성
- 구조 및 구성
- 완성도
- 유용성
```
