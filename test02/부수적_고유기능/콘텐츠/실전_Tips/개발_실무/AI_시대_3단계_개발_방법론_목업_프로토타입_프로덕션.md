# AI 시대의 3단계 개발 방법론: 목업 → 프로토타입 → 프로덕션

> AI를 활용한 소프트웨어 개발에서 목업과 프로토타입의 경계가 사라지고 있습니다. 각 단계의 본질을 이해하고 AI 시대에 맞는 개발 전략을 수립합니다.

---

## 3단계 개발 흐름

```
┌─────────────┐
│   목업      │  기능 구현 + 구조 설계
│  (Mock-up)  │  "일단 돌아가게 만든다"
└──────┬──────┘
       │ + 디자인 시스템 적용
       ↓
┌─────────────┐
│ 프로토타입   │  목업 + 세련된 디자인
│ (Prototype) │  "보여줄 수 있게 만든다"
└──────┬──────┘
       │ + 체계적 개발 방법론
       ↓
┌─────────────┐
│  프로덕션    │  프로토타입 + 유지보수성
│ (Production)│  "오래 쓸 수 있게 만든다"
└─────────────┘
```

---

## 목업 (Mock-up)

목업은 기능이 돌아가는 최소한의 형태입니다. 디자인은 신경 쓰지 않습니다. 버튼을 누르면 동작하고, 데이터가 저장되고, 화면이 전환됩니다.

집을 짓기 전 골조를 세우는 것과 같습니다. 벽지나 인테리어 없이 기둥과 벽만 세웁니다. 공간 구조와 동선이 맞는지 확인합니다.

목업 단계의 질문:
- 이 기능이 필요한가?
- 사용자 흐름이 자연스러운가?
- 데이터 구조가 맞는가?

AI와 함께 개발하면 목업 단계가 빨라집니다. 대화하면서 실시간으로 코드가 만들어집니다. "회원가입 기능 만들어줘"라고 하면 몇 분 안에 작동하는 코드가 나옵니다.

---

## 프로토타입 (Prototype)

프로토타입은 목업에 디자인을 입힌 것입니다. 색상, 폰트, 여백, 애니메이션이 적용됩니다. 실제 서비스처럼 보입니다.

집에 비유하면 인테리어를 마친 상태입니다. 벽지를 바르고, 조명을 설치하고, 가구를 배치합니다. 손님에게 보여줄 수 있는 상태입니다.

프로토타입 단계의 질문:
- 사용하기 편한가?
- 시각적으로 매력적인가?
- 브랜드와 일관성이 있는가?

AI 시대에는 목업과 프로토타입의 경계가 흐려집니다. 처음부터 디자인 시스템을 적용하면 목업이 곧 프로토타입이 됩니다.

---

## 프로덕션 (Production)

프로덕션은 실제 사용자에게 서비스되는 최종 버전입니다. 프로토타입에 체계적인 개발 방법론이 더해집니다.

집에 비유하면 입주 후 10년을 살 집입니다. 배관이 제대로 되어 있어야 하고, 전기 배선이 안전해야 합니다. 나중에 고장 나면 수리할 수 있어야 합니다.

프로덕션 단계에서 추가되는 것들:
- 에러 처리와 예외 상황 대응
- 보안 강화 (인증, 권한, 데이터 보호)
- 성능 최적화 (로딩 속도, 응답 시간)
- 모니터링과 로깅
- 테스트 코드
- 문서화

프로덕션의 핵심은 유지보수성입니다. 1년 후에 코드를 열어봐도 이해할 수 있어야 합니다. 다른 개발자가 이어받아도 수정할 수 있어야 합니다.

---

## AI가 바꾼 개발 프로세스

전통적인 개발 프로세스는 순차적이었습니다.

```
기획서 작성 → 디자인 → 개발 → 테스트 → 배포
```

각 단계가 완료되어야 다음 단계로 넘어갔습니다. 기획서 없이 디자인을 할 수 없고, 디자인 없이 개발을 시작할 수 없었습니다.

AI와 함께하면 이 과정이 동시에 진행됩니다.

```
대화하면서 기능 설명 + 코드 생성 + 즉시 테스트
```

"회원가입 페이지 만들어줘. 이메일 인증 필요하고, 비밀번호는 8자 이상이어야 해."

이 한 문장으로 기획, 설계, 구현이 동시에 일어납니다. 결과물을 보고 "여기에 소셜 로그인도 추가해줘"라고 하면 바로 수정됩니다.

---

## 각 단계의 본질

| 단계 | 본질 | 핵심 질문 |
|------|------|----------|
| 목업 | 구조와 기능 검증 | 이 흐름이 맞는가? |
| 프로토타입 | 사용자 경험 검증 | 이 디자인이 좋은가? |
| 프로덕션 | 장기 운영 준비 | 10년 후에도 유지보수할 수 있는가? |

목업에서 프로토타입으로 가는 것은 옷을 입히는 것입니다.
프로토타입에서 프로덕션으로 가는 것은 건강검진을 받는 것입니다.

겉보기에 멀쩡해 보여도 내부에 문제가 있을 수 있습니다. 에러 처리가 없으면 예상치 못한 상황에서 서비스가 멈춥니다. 보안이 취약하면 해킹당합니다. 테스트가 없으면 수정할 때마다 다른 곳이 망가집니다.

---

## 실전 적용 전략

### 1. 빠르게 목업부터 만든다

완벽한 기획서를 쓰느라 시간 낭비하지 않습니다. AI에게 핵심 기능을 설명하고 바로 코드를 받습니다. 돌아가는 것을 보면서 기획을 다듬습니다.

### 2. 디자인 시스템을 미리 정한다

색상, 폰트, 버튼 스타일을 미리 정해두면 목업이 곧 프로토타입이 됩니다. "이 디자인 시스템 적용해서 만들어줘"라고 하면 처음부터 예쁜 코드가 나옵니다.

### 3. 프로덕션 체크리스트를 사용한다

프로토타입을 프로덕션으로 올리기 전에 체크리스트를 돌립니다. 에러 처리, 보안, 성능, 테스트, 문서화. 하나라도 빠지면 프로덕션이 아닙니다.

### 4. 단계를 건너뛰지 않는다

아무리 급해도 목업 없이 바로 프로덕션을 만들면 안 됩니다. 기초 공사 없이 인테리어부터 하면 나중에 집이 무너집니다.

## 기능은 단계별로 발견된다

```
목업 단계:
"이 정도면 작동하겠지" (70~80% 기능)

프로토타입 단계:
"아, 이것도 필요하네!" (누락 기능 발견)
→ 검색, 필터, Export, 알림 등 추가

프로덕션 단계:
"사용자가 이것도 원하네!" (추가 기능 발견)
→ 고급 통계, 통합, 자동화 등 추가
```

모든 기능을 처음부터 구현하지 않는 이유:
1. **시간 낭비 방지**: 안 쓰이는 기능 만들 수 있음
2. **빠른 검증**: 핵심만 먼저 검증
3. **사용자 피드백**: 실제 사용 중 필요 기능 발견
4. **점진적 완성**: 단계별로 안정적 추가

## AI 시대의 결정적 차이

**과거 (AI 없음):**
```
목업: 껍데기, 작동 안 함
  ↓ 막대한 시간/비용
프로덕션: 실제 작동

→ 목업은 "보여주기용"
```

**현재 (AI 활용):**
```
목업: 실제 작동, 프로덕션급 기능
  ↓ 디자인/체계만 추가
프로덕션: 목업 + α

→ 목업도 "실제 사용 가능"
```

과거에는 목업이 그림이었습니다. 지금은 목업이 작동하는 제품입니다.

---

## 단계 전환 타이밍

### 목업 → 프로토타입 전환 시점
- 목업으로 핵심 기능 검증 완료
- 사용자가 "이것도 있으면 좋겠다" 피드백 시작
- 베타 서비스 출시 계획
- 투자 유치 준비

### 프로토타입 → 프로덕션 전환 시점
- 프로토타입으로 대부분 기능 검증 완료
- 정식 서비스 출시 확정
- 팀 확장 예정
- 장기 운영 계획

---

## 결론

**AI 시대의 목업 = 프로덕션급 기능을 가진 작동하는 제품**

3단계의 차이는:
1. **디자인 시스템의 정교함** (프로토타입 단계에서 추가)
2. **관리 방법론의 체계성** (프로덕션 단계에서 추가)
3. **기능의 완성도** (각 단계에서 점진적 발견 및 추가)

단계를 이해하면 언제 무엇을 해야 하는지 알게 됩니다. 목업에서 디자인을 고민하지 않습니다. 프로토타입에서 유지보수를 걱정하지 않습니다. 각 단계에 집중하면 전체가 완성됩니다.

## SSALWorks와의 연결

SSALWorks의 SAL Grid는 이 3단계를 체계적으로 관리하는 도구입니다.

- Stage 1~2: 개발 준비 + 목업 수준의 기능 구현
- Stage 3~4: 프로토타입 수준의 완성도
- Stage 5: 프로덕션 배포 및 안정화

각 Stage를 통과할 때마다 Stage Gate 검증을 거칩니다. 목업이 충분한지, 프로토타입이 사용 가능한지, 프로덕션이 안정적인지 확인합니다.

도구가 바뀌어도 방법론은 그대로입니다. 목업 → 프로토타입 → 프로덕션. 이 흐름을 이해하면 어떤 프로젝트든 체계적으로 진행할 수 있습니다.
