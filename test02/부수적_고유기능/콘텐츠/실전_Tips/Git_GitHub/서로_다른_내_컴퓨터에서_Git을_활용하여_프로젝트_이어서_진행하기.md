# 서로 다른 내 컴퓨터에서 Git을 활용하여 프로젝트 이어서 진행하기

## 왜 Google Drive 동기화 대신 Git을 사용해야 하는가

개발 작업을 할 때 여러 컴퓨터에서 작업해야 하는 경우가 많습니다. 가정에서는 데스크톱으로, 외출 시에는 노트북으로 작업하는 경우가 대표적입니다. 이때 Google Drive나 Dropbox 같은 클라우드 동기화 서비스를 사용하고 싶은 유혹이 생깁니다. 하지만 Claude Code와 함께 개발할 때는 이 방법이 심각한 문제를 일으킬 수 있습니다.

Google Drive 동기화가 문제가 되는 이유는 Claude Code의 작업 방식 때문입니다. Claude Code는 세션 동안 수많은 임시 파일과 캐시 파일을 생성하고 수정합니다. 이 파일들이 Google Drive와 동기화되면 충돌이 발생하거나, 동기화 지연으로 인해 파일이 손상될 수 있습니다. 특히 여러 파일이 동시에 수정될 때 동기화 타이밍이 어긋나면 프로젝트 전체가 꼬여버리는 상황이 발생합니다.

SSALWorks 방법론에서는 이런 문제를 Git을 통해 해결합니다. Git은 명시적으로 커밋하고 푸시할 때만 변경사항이 반영되므로, 작업 중인 불완전한 상태가 다른 컴퓨터로 전파되지 않습니다. 개발 작업의 안정성을 위해 클라우드 동기화 대신 Git을 사용하는 것이 훨씬 안전한 선택입니다.

## 동일 경로 설정의 중요성

두 컴퓨터에서 Git으로 작업할 때 가장 중요한 것은 프로젝트 경로를 동일하게 유지하는 것입니다. 예를 들어 컴퓨터 A에서 프로젝트가 `C:\Projects\MyApp`에 있다면, 컴퓨터 B에서도 정확히 같은 경로에 프로젝트를 위치시켜야 합니다. 경로가 다르면 Claude Code의 CLAUDE.md나 규칙 파일에 기록된 절대 경로들이 작동하지 않을 수 있기 때문입니다.

Windows 사용자라면 두 컴퓨터 모두에서 같은 드라이브 문자를 사용하는 것이 좋습니다. Mac 사용자라면 홈 디렉토리 기준으로 동일한 상대 경로를 유지하세요. 이렇게 하면 Git 저장소를 클론할 때 추가적인 경로 수정 없이 바로 작업을 시작할 수 있습니다.

## 초기 설정 방법

새 컴퓨터에서 처음 작업을 시작할 때는 Git 클론을 수행합니다. GitHub에 프로젝트가 이미 올라가 있다면 단 한 번의 클론으로 모든 파일과 히스토리를 가져올 수 있습니다. 터미널을 열고 원하는 위치에서 `git clone` 명령어를 실행하면 됩니다. 이 작업은 컴퓨터마다 최초 한 번만 하면 됩니다.

클론이 완료되면 Claude Code의 설정 파일들도 모두 가져와지므로, 기존 컴퓨터에서 하던 것처럼 바로 작업을 이어갈 수 있습니다. CLAUDE.md, 규칙 파일들, work_logs까지 모든 것이 그대로 유지됩니다. SSALWorks 방법론의 모든 설정이 Git을 통해 함께 관리되기 때문입니다.

## 일상적인 작업 워크플로우

두 컴퓨터 사이를 오가며 작업할 때는 간단한 규칙만 따르면 됩니다. 컴퓨터를 바꿔서 작업을 시작할 때마다 먼저 `git pull`을 실행합니다. 이렇게 하면 다른 컴퓨터에서 마지막으로 푸시한 변경사항을 가져옵니다. 이 단계를 빠뜨리면 충돌이 발생할 수 있으니 습관처럼 실행해야 합니다.

작업이 끝나고 컴퓨터를 끄기 전에는 반드시 변경사항을 커밋하고 푸시합니다. 커밋 메시지는 간단하게라도 무엇을 했는지 기록해두면 나중에 다른 컴퓨터에서 확인할 때 유용합니다. 푸시까지 완료해야 GitHub에 변경사항이 저장되고, 다른 컴퓨터에서 pull로 가져올 수 있습니다.

Claude Code로 작업하면서 커밋과 푸시를 요청할 수도 있습니다. "지금까지 작업 커밋해줘"라고 말하면 Claude Code가 적절한 커밋 메시지와 함께 Git 작업을 대신 해줍니다. 이 방법을 활용하면 Git 명령어를 직접 입력하지 않아도 됩니다.

## 핵심 Git 명령어 정리

자주 사용하는 Git 명령어는 네 가지입니다. `git pull`은 원격 저장소의 최신 변경사항을 가져옵니다. `git add .`은 변경된 모든 파일을 스테이징 영역에 추가합니다. `git commit -m "메시지"`는 스테이징된 변경사항을 로컬에 저장합니다. `git push`는 로컬의 커밋들을 원격 저장소에 업로드합니다.

매일 작업 시작과 끝에 이 명령어들을 실행하는 습관을 들이면 두 컴퓨터 사이의 동기화가 자연스럽게 유지됩니다. 실수로 pull을 잊고 작업을 시작했더라도 걱정하지 마세요. Git은 충돌을 감지하고 해결할 수 있는 도구를 제공합니다.

## 문제 상황 대처법

가끔 pull 없이 작업을 시작해서 충돌이 발생할 수 있습니다. 이때는 당황하지 말고 Git이 표시하는 충돌 메시지를 확인하세요. 대부분의 경우 어떤 파일에서 충돌이 발생했는지 알려주며, 두 버전 중 어느 것을 선택할지 결정하면 됩니다. Claude Code에게 "Git 충돌 해결해줘"라고 요청하면 도움을 받을 수도 있습니다.

충돌을 피하는 가장 좋은 방법은 작업 시작 전 반드시 pull을 하는 습관을 들이는 것입니다. 또한 하나의 컴퓨터에서 작업을 완전히 마무리하고 푸시한 후에 다른 컴퓨터로 이동하는 것이 좋습니다. 두 컴퓨터에서 동시에 작업하는 상황은 가능한 피해야 합니다.

## SSALWorks와 Git의 시너지

SSALWorks 방법론은 Git과 잘 어울리도록 설계되어 있습니다. 프로젝트의 모든 설정 파일, 규칙 파일, 작업 로그가 Git으로 관리되므로 어느 컴퓨터에서든 동일한 개발 환경을 유지할 수 있습니다. Task를 완료할 때마다 커밋하면 작업 히스토리가 자연스럽게 남고, 필요할 때 이전 상태로 돌아갈 수도 있습니다.

Pre-commit Hook을 설정해두면 커밋할 때마다 자동으로 파일 동기화나 검증이 이루어집니다. 이렇게 Git의 기능과 SSALWorks의 자동화가 결합되면 여러 컴퓨터에서 작업하더라도 일관된 개발 프로세스를 유지할 수 있습니다. 클라우드 동기화의 불안정성 대신 Git의 확실한 버전 관리를 선택하세요.
