# Task를 적절한 크기로 나누는 기준

> 이 문서는 프로젝트 Task를 적절한 크기로 분할하는 기준을 설명합니다.

---

## 왜 Task를 나누는가

| 이점 | 설명 |
|------|------|
| 관리 용이 | 진행 상황 추적 쉬움 |
| 병렬 작업 | 여러 Task 동시 진행 |
| 품질 보장 | 각 Task별 검증 가능 |
| 리스크 감소 | 실패 범위 제한 |

---

## 적절한 Task 크기

| 크기 | 권장 |
|------|------|
| 1시간 이내 | 이상적 |
| 2-4시간 | 적절함 |
| 하루 이상 | 더 나눠야 함 |

---

## 분할 기준

| 기준 | 예시 |
|------|------|
| 기능 단위 | 로그인, 결제, 알림 |
| 레이어 단위 | UI, API, DB |
| 화면 단위 | 로그인 페이지, 대시보드 |

---

## 분할 예시: 로그인 기능

```
너무 큰 Task:
- "로그인 기능 전체 구현"

적절하게 분할:
1. S2F1: 로그인 페이지 UI
2. S2F2: 로그인 폼 유효성 검사
3. S2BA1: 로그인 API 엔드포인트
4. S2S1: OAuth 콜백 처리
5. S2T1: 로그인 테스트
```

---

## Task 분할 요청

```
"이 기능을 Task로 나눠줘"
"각 Task가 2-4시간 내에 완료될 수 있게 분할해줘"
```

---

## 분할 확인 기준

| 확인 항목 | 질문 |
|----------|------|
| 독립성 | 다른 Task 없이 완료 가능한가? |
| 검증 가능 | 완료 여부를 확인할 수 있는가? |
| 명확성 | 무엇을 해야 하는지 명확한가? |

---

## 너무 작은 Task 피하기

```
나쁜 예:
- "버튼 색상 변경"
- "텍스트 수정"

좋은 예:
- "로그인 페이지 스타일 개선"
```

---

## Task 의존성 관리

```
S2F1 (UI) → S2BA1 (API) → S2T1 (테스트)

의존성이 있으면 순서대로 진행
```
