# 작성자와 검증자를 다른 서브에이전트로 분리하여 객관적으로 검증하기

> 이 문서는 작성자와 검증자를 분리해야 하는 이유와 서브에이전트 검증 방법을 설명합니다.

---

## 왜 별도 검증이 필요한가

같은 사람(또는 AI)이 작성하고 검증하면 맹점이 생깁니다. 자신이 작성한 코드는 "당연히 맞다"고 생각하기 때문입니다. 다른 시각으로 봐야 문제를 발견할 수 있습니다.

```
자체 검증 (잘못된 방식):
backend-developer가 작성 → backend-developer가 검증 → 맹점 발생

별도 검증 (올바른 방식):
backend-developer가 작성 → code-reviewer가 검증 → 문제 발견
```

이 원칙은 CLAUDE.md 절대 규칙 2번에 명시되어 있습니다: **작성자 ≠ 검증자**

## 작업 에이전트와 검증 에이전트 매칭

| Area | 작업 내용 | Task Agent | Verification Agent |
|------|----------|------------|-------------------|
| F | 프론트엔드 | frontend-developer | code-reviewer, qa-specialist |
| BA | 백엔드 API | backend-developer | code-reviewer |
| S | 보안/인증 | security-specialist | security-auditor |
| D | 데이터베이스 | database-specialist | database-specialist* |
| BI | 백엔드 인프라 | backend-developer | code-reviewer |
| E | 외부 연동 | backend-developer | code-reviewer |

*D Area는 별도 database-specialist 인스턴스가 검증합니다.

## Claude Code에게 요청하기

### 작업 후 검증 요청

```
"backend-developer가 작성한 subscription-cancel.js를
code-reviewer로 검증해줘:
- 에러 처리 적절한지
- 보안 이슈 없는지
- 코드 컨벤션 맞는지"
```

### 보안 관련 검증 요청

```
"security-specialist가 작성한 google-callback.js를
security-auditor로 검증해줘"
```

### 프론트엔드 검증 요청

```
"frontend-developer가 작성한 login.html을
qa-specialist로 검증해줘"
```

## 금지 행동

### 자체 검증 금지

```
잘못된 방식: "backend-developer로 API 만들고 검증도 해줘"

올바른 방식: "backend-developer로 API 만들어줘"
(완료 후)
"code-reviewer로 검증해줘"
```

### Main Agent 직접 검증 금지

Main Agent가 직접 "괜찮아 보입니다" 판단하면 안 됩니다. 반드시 서브에이전트(code-reviewer 등)를 호출해서 검증합니다.

### 검증 생략 금지

아무리 간단한 수정이라도 검증 에이전트를 투입합니다.
