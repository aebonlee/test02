# 작업 중 중요한 결정사항과 변경사항을 수시로 문서화하기

> 이 문서는 작업 중 중요한 내용을 수시로 문서화하는 방법을 설명합니다.

---

## 왜 수시로 문서화하는가

작업 중에 발생한 중요한 정보는 시간이 지나면 잊어버립니다. 설정 방법, 트러블슈팅 과정, 결정 사항 등은 나중에 다시 필요할 때 찾을 수 없으면 처음부터 다시 해야 합니다.

```
나중에 문서화 (잘못된 방식):
"나중에 정리해야지" → 잊어버림 → 다음에 같은 문제 발생 → 처음부터 다시

즉시 문서화 (올바른 방식):
작업 완료 → 바로 문서화 → 다음에 같은 문제 발생 → 문서 참고 → 빠르게 해결
```

---

## 문서화 대상

| 종류 | 예시 |
|------|------|
| 설정 방법 | Supabase OAuth 설정, Vercel 환경변수, API 키 발급 과정 |
| 트러블슈팅 | 오류 발생 → 원인 분석 → 해결 과정 |
| 중요 결정 | 기술 선택 이유, 아키텍처 결정, 라이브러리 선택 이유 |
| 외부 서비스 | 가맹점 등록, 계정 생성, 연동 절차 |

---

## Claude Code에게 요청하기

### 설정 과정 문서화

```
"방금 한 Supabase OAuth 설정 과정 문서화해줘"

"Google Cloud Console에서 한 작업 순서대로 정리해줘"
```

### 트러블슈팅 문서화

```
"CORS 오류 해결 과정을 문서화해줘:
- 증상
- 원인
- 해결 방법"
```

### 결정 사항 문서화

```
"Resend를 이메일 서비스로 선택한 이유 문서화해줘"
```

## 저장 위치

| 위치 | 내용 |
|------|------|
| `.claude/work_logs/current.md` | 작업 기록, 일일 로그 |
| `S?_*/Documentation/` | 기능 명세, API 문서, 설정 가이드 |
| `부수적_고유기능/콘텐츠/Tips/` | 재사용 가능한 팁 |

### work_logs 형식

```markdown
## 2025-12-20

### 완료 작업
- S2BA1: 구독 취소 API 구현

### 트러블슈팅
- CORS 오류 발생 → vercel.json에 헤더 추가로 해결

### 다음 작업
- S2BA2: 구독 재개 API 구현
```

---

## 주의사항

**작업 직후 바로 문서화**: 나중에 미루면 세부사항을 잊습니다.

**민감정보 제외**: API 키, 비밀번호 등은 문서에 직접 포함하지 않습니다.

```
잘못된 방식: API_KEY=sk_live_abc123...
올바른 방식: API 키는 Vercel 환경변수에 설정됨 (RESEND_API_KEY)
```

**구체적으로 작성**: "설정했음"이 아니라 어떻게 설정했는지 단계별로 기록합니다.
