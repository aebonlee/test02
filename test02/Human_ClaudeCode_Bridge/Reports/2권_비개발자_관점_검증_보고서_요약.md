## 주요 문제점 (반드시 개선 필요!)

### 1. 용어 과부하 ⚠️ **높은 심각도**

**문제**:
- 01편 섹션 3에 **30개 이상** 새로운 용어가 한꺼번에 나열
- 입문자가 한 번에 소화 가능한 개념은 5~7개

**구체적 사례**:
```
01편 130~246줄:
브라우저, HTML, CSS, JS, DB, 서버, 클라우드, 도메인, 호스팅,
프레임워크, 라이브러리, API, Git, 배포, UI/UX, CMS, SPA,
반응형, HTTPS, SSL/TLS, 캐시, CDN
→ 30개가 100줄 안에 쏟아짐
```

**입문자 반응**: "머리가 터질 것 같아요..."

**해결 방법**:
- 이 섹션을 3개 편으로 분리 OR
- "지금 알아야 할 5개" vs "나중에 알아도 될 용어"로 우선순위 구분

---

### 2. 약어 설명 불충분 ⚠️ **높은 심각도**

**문제**:
- HTML, CSS, DOM 같은 약어의 영문 풀네임만 제공
- 각 단어가 무슨 뜻인지 설명 없음

**구체적 사례**:

| 약어 | 현재 설명 | 문제 | 개선안 |
|------|----------|------|--------|
| HTML | HyperText Markup Language | HyperText? Markup? 뭔데? | HyperText(링크로 연결된 텍스트) + Markup(태그로 의미 표시) |
| CSS | Cascading Style Sheets, 종속형 시트 | '종속형'이 더 어려움 | 스타일시트. Cascading(위→아래 적용)은 나중에 알아도 됨 |
| DOM | Document Object Model, 객체 모델 | '객체 모델'이 더 어려움 | 브라우저가 HTML을 해석한 내부 구조 (지금은 몰라도 됨) |

**입문자 반응**: "설명을 설명해야 하는 상황..."

---

### 3. 학습 동기 불명확 ⚠️ **중간 심각도**

**문제**:
- "왜 이걸 배워야 하나?"에 대한 답이 약함
- 이론적 장점보다 실용적 필요성이 중요

**구체적 사례**:

| 편 | 현재 동기 제시 | 부족한 점 | 개선안 |
|----|--------------|----------|--------|
| 03편 (CLI) | "빠르고 강력함" | 입문자에게는 마우스가 더 편해 보임 | "Git, npm, Node.js는 모두 CLI로만 작동. CLI 없이는 개발 시작 불가" |
| 08편 (Language) | "웹사이트의 구조/스타일/동작" | 추상적 | "모든 웹사이트는 HTML/CSS/JS로 만들어짐. 이 3가지 없이는 개발자 불가" |

**입문자 반응**: "왜 배우는지 모르니까 집중이 안 돼요"

---

### 4. 전제 지식 가정 ⚠️ **중간 심각도**

**문제**:
- 일부 개념을 "이미 알고 있다" 전제로 설명

**구체적 사례**:

**사례 1**: DNS 설명 (02편)
```
"여러 DNS 서버를 거쳐 최종적으로 IP 주소를 찾아냄"
```
→ 입문자 반응: "왜 여러 번 거쳐야 해? 한 번에 안 돼?"

**개선안**: 첫 설명은 "도메인 → IP 주소"로 단순화. 계층 구조는 "더 알아보기" 박스로 분리

**사례 2**: Git 스테이징 영역 (05편)
```
"커밋할 변경사항을 임시로 저장하는 공간"
```
→ 입문자 반응: "왜 중간 단계가 필요해? 그냥 바로 저장하면 안 돼?"

**개선안**: "10개 파일을 수정했는데 5개만 먼저 커밋하고 싶을 때, 스테이징으로 선택할 수 있습니다" (실용적 예시)
