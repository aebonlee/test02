Task 선정 매트릭스

작성일: 2025-11-26 수정일: 2025-12-12 버전: v4.0 용도: Stage×Area 기반 Task 선정 가이드 (5×11 매트릭스)

1. 개요

이 매트릭스는 PROJECT SAL GRID에서 Task를 선정할 때 사용하는 실전 가이드입니다.

중요 변경사항 (v4.0):

  • 기존 S1-S6 (6단계) → **S1-S5 (5단계)**로 변경
  • P1-P3 (Preliminary)는 GRID 범위 밖에서 별도 관리
  • S1 프로토타입이 제거됨 (P3로 이동)

5×11 매트릭스와의 관계:

  • 5×11 매트릭스: 이론적 참고자료 (55개 영역 가능)
  • Task 선정 매트릭스: 실전 적용 (Stage별로 필요한 Area만 선별)

4. Task 선정 원칙 (3대 원칙)

원칙 1: 파일 1개 = Task 1개 (기본)

생성되는 파일 하나당 Task 하나를 만듭니다.

올바른 예시:

S2F1: src/app/page.tsx 생성
S2F2: src/app/search/page.tsx 생성
S2F3: src/components/Header.tsx 생성

잘못된 예시:

S2F1: 메인 페이지 전체 구현
  - page.tsx, Hero.tsx, Features.tsx...
→ 너무 크고 독립 작업 불가능

원칙 2: 동시 작업 가능한 유사 작업 통합

같은 기능의 세부 파일들은 하나의 Task로 묶습니다.

올바른 예시:

S2F5: 프로필 관련 컴포넌트
  - ProfileCard.tsx
  - ProfileEdit.tsx
  - ProfileAvatar.tsx
→ 같은 "프로필" 기능, 동시 작업 가능

잘못된 예시:

S2F5: ProfileCard.tsx
S2F6: ProfileEdit.tsx
S2F7: ProfileAvatar.tsx
→ 너무 세분화, 관리 오버헤드

원칙 3: 의존성 순환 금지

Task 간 의존성에서 순환이 발생하면 안 됩니다.

잘못된 예시:

A → B → C → A (순환 발생)

올바른 예시:

A → B → C (단방향)

순환 발견 시 의존성 구조를 수정하세요.

5. 판단 기준

언제 분리?

  • 독립적인 기능
  • 다른 개발자가 동시 작업 불가
  • 의존성이 복잡함

예시:

  • 로그인 페이지 vs 회원가입 페이지 → 분리!
  • Header vs Footer → 분리!

언제 통합?

  • 같은 기능의 세부 파일들
  • 함께 수정될 가능성 높음
  • 동시 작업 가능

예시:

  • 버튼 컴포넌트 variants → 통합!
  • 프로필 관련 컴포넌트들 → 통합!

7. Gate 체크리스트

P3 → S1 Gate (프로토타입 → 개발준비)

  • 프로토타입 핵심 기능 작동
  • 사용자 피드백 반영
  • 개발 범위 확정

S1 → S2 Gate (개발준비 → 개발1차)

  • 개발 환경 구축 완료
  • Git 저장소 설정
  • Supabase 프로젝트 설정
  • CI/CD 파이프라인 구축

S2 → S3 Gate (개발1차 → 개발2차)

  • 인증 시스템 완료
  • 핵심 페이지 구현 완료
  • 기본 API 작동
  • 기본 테스트 통과

S3 → S4 Gate (개발2차 → 개발3차)

  • 고급 기능 구현 완료
  • 외부 서비스 연동 완료
  • 통합 테스트 가능 상태

S4 → S5 Gate (개발3차 → 운영)

  • 모든 테스트 통과 (Unit, E2E)
  • 성능 기준 충족
  • 보안 검증 완료
  • 프로덕션 배포 준비 완료
