## Order Sheet란?

**Order Sheet는 Claude Code에게 작업을 지시하는 표준화된 문서입니다.**

명확하고 구조화된 지시를 통해 더 정확한 결과를 얻을 수 있습니다.

---

## 📋 Order Sheet ID 체계

### ID 형식
```
OS-[TYPE]-[STAGE]-[ITEM]-[SEQ]
```

| 구분 | 설명 | 예시 |
|------|------|------|
| OS | Order Sheet 접두사 | OS |
| TYPE | 단계 유형 | SP(특별), PR(준비), S1-S5(실행) |
| STAGE | 영역 코드 | 01~11 |
| ITEM | 작업 항목 | A, B, C... |
| SEQ | 순번 | 001, 002... |

### 예시
- `OS-SP-01-A-001`: 특별단계 - 디렉토리 구조 생성 - 첫 번째 Order
- `OS-PR-02-B-001`: 준비단계(사업계획) - 경쟁분석 - 첫 번째 Order
- `OS-S1-03-A-001`: Stage 1(프로토타입) - Frontend - 첫 번째 Order

## ✍️ 좋은 Order Sheet 작성 팁

### 1. 명확한 목표 설정

**❌ 나쁜 예:**
```
"로그인 기능 만들어주세요"
```

**✅ 좋은 예:**
```
"JWT 기반 로그인 기능을 구현해주세요.
- 토큰 만료 시간: 24시간
- 리프레시 토큰 포함
- 비밀번호는 bcrypt로 해싱
- 실패 시 5회 제한"
```

---

### 2. 구체적인 기술 스택 명시

**❌ 나쁜 예:**
```
"API 만들어주세요"
```

**✅ 좋은 예:**
```
"Express.js로 REST API를 만들어주세요.
- TypeScript 사용
- 에러 핸들링 미들웨어 포함
- Swagger 문서 자동 생성
- 응답 형식: JSON"
```

### 3. 참조 자료 제공

**포함할 내용:**
- 기존 코드가 있다면 파일 경로 명시
- 참고할 디자인이나 문서 링크
- 유사한 기존 기능 예시

**예시:**
```
## 참조 자료
- 기존 회원가입 API: `src/api/auth/signup.ts`
- UI 디자인: `design/mockup/login.png`
- 참고 문서: flow.md의 User Flow #1
```

## 📂 템플릿 사용하기

### 자동 템플릿 로드

1. 좌측 사이드바에서 작업 단계 클릭
2. 해당 단계에 맞는 템플릿이 자동 로드
3. 템플릿 내용을 수정하여 사용

### 템플릿 종류

**특별단계 템플릿:**
- 디렉토리 구조 생성
- SAL GRID 생성

**준비단계 템플릿:**
- 사업계획 (시장조사, 경쟁분석, 사업계획서)
- 프로젝트 기획 (Plan, Requirements, Architecture, UI/UX, DB, API)

**실행단계 템플릿:**
- S1: 프로토타입 제작
- S2: 개발 준비
- S3: 개발 1차
- S4: 개발 2차
- S5: 개발 마무리

## 💾 Order Sheet 저장 및 전송

### 저장 방법

1. Order Sheet 작성 완료
2. **"Download Order Sheet"** 버튼 클릭
3. 자동으로 `프로젝트명/Web_ClaudeCode_Bridge/Inbox/` 폴더에 저장

### 파일 형식
- 파일명: `ordersheet_YYYY-MM-DD_HH-mm-ss.json`
- 인코딩: UTF-8
- 프로젝트 ID 자동 포함

### Claude Code 연동

```
1. Order Sheet 저장
   ↓
2. Inbox/ 폴더에 파일 생성
   ↓
3. 터미널에서 `claude` 명령어 실행
   ↓
4. Claude Code가 Inbox/ 모니터링
   ↓
5. 새 Order Sheet 감지
   ↓
6. 작업 시작
   ↓
7. 완료 후 Outbox/에 결과 저장
   ↓
8. Dashboard에서 "Load from Outbox" 클릭
   ↓
9. 결과 확인
```

## ⚠️ 주의사항

### 필수 입력 항목
- ✅ 작업 개요
- ✅ 상세 요구사항 (최소 1개)
- ✅ 예상 결과물

### 금지 사항
- ❌ 민감한 정보 (API 키, 비밀번호 등) 직접 입력 금지
- ❌ 실행 불가능한 요구사항
- ❌ 모호한 지시 ("좋게 만들어주세요" 등)

### 권장 사항
- ✅ 한 Order Sheet에 하나의 주요 작업만 포함
- ✅ 복잡한 작업은 여러 Order Sheet로 분리
- ✅ 이전 작업 결과를 참조할 때는 Order ID 명시
- ✅ 테스트 요구사항 명시

## 📚 Order Sheet 예시

### 예시 1: 회원가입 API 구현

```markdown
# Order Sheet

## 기본 정보
- **Order ID**: OS-S3-02-A-001
- **Project ID**: A3B5C7D9-P001
- **작성일**: 2025-12-01
- **우선순위**: 높음
- **예상 소요시간**: 4시간

## 작업 개요
회원가입 API를 구현합니다.
이메일, 비밀번호, 닉네임, 실명을 입력받아
Supabase Auth와 users 테이블에 저장합니다.

## 상세 요구사항
1. POST /api/auth/signup 엔드포인트 생성
2. 입력값 검증 (이메일 형식, 비밀번호 강도, 닉네임 중복)
3. Supabase Auth로 계정 생성
4. users 테이블에 추가 정보 저장 (user_id 자동 생성)
5. 에러 핸들링 (중복, 형식 오류 등)
6. 성공 시 user_id 반환

## 참조 자료
- User Flow #1: flow.md
- DB 스키마: users 테이블 구조
- 기존 로그인 API: src/api/auth/login.ts

## 예상 결과물
생성될 파일:
- src/api/auth/signup.ts
- src/api/auth/signup.test.ts
- src/utils/generateUserId.ts

수정될 파일:
- src/routes/auth.ts (signup 라우트 추가)

## 제약 조건
- TypeScript 사용
- Express.js 프레임워크
- Supabase SDK 사용
- 비밀번호는 평문 저장 금지

## 추가 지시사항
- Unit test 포함
- API 응답 형식 통일 (success, data, message)
- 에러 응답 코드: 400 (검증 실패), 409 (중복), 500 (서버 오류)
```
