> **용도:** Production/index.html의 서비스 소개 페이지
> **구조:** 7개 섹션 통합 구성

## 1. 개요

**SSAL Works**는 AI를 활용해서 SAL Grid 등의 체계적 개발 방법론을 적용하여 풀스택 웹사이트를 개발하는 플랫폼입니다.

- **AI란?**
  - Claude Code: Anthropic이 만든 AI 코딩 도구로, 사람 대신 코딩을 수행하며, 주력 AI입니다.
  - Gemini CLI: Google에서 개발한 AI 모델로, 비용 효율적인 분석, 코드 리뷰, CLI 환경에서의 지시 이행 등 보조적인 역할을 수행합니다.
- **SAL Grid란?**
  - SAL(Stage, Area, Level의 3차원 좌표계) 기반의 Grid를 활용하여 개발 프로젝트의 실행/관리/통제/검증을 진행하는 시스템입니다.
- **풀스택 웹사이트란?**
  - 프론트엔드(화면) + 백엔드(서버/API) + 데이터베이스를 모두 갖춘 완전한 웹사이트입니다.
- **SSAL**: S(Sunny) + SAL(Stage, Area, Level의 3차원 좌표계). 한글 발음 "쌀"은 핵심 먹을거리를 의미합니다. 황금쌀 방앗간처럼, 아이디어를 가져와서 돈 버는 풀스택 웹사이트를 만드는 곳입니다.

<span style="font-size: 12px;">개발기술 관련 지식이 없어도, 아이디어, 염원, 의지만 있으면 실제 비즈니스에 사용할 수 있는 완전한 풀스택 웹사이트를 만들 수 있습니다.</span>

**누구를 위한 플랫폼인가?**

| 당신이 이런 상황이라면 | SSAL Works가 해결합니다 |
|----------------------|----------------------|
| **스타트업**: MVP 만들려는데 개발자 채용할 돈도 시간도 없다 | 1개월 안에 직접 만들고 런칭 |
| **직장인 부업**: 퇴근 후 시간을 활용해서 뭔가 만들고 싶다 | 매일 조금씩 → 완성된 풀스택 웹사이트 |
| **시니어**: 경험은 많은데 기술을 모른다 | 개발기술 관련 지식이 없어도 AI가 코딩을 대신 수행 |
| **기존 사업자**: 외주 맡기면 수천만원~수억원인데... | 300만원 + 직접 컨트롤 가능 |

**바이브 코딩의 한계**

**바이브 코딩(Vibe Coding)**이란 AI에게 "이거 만들어줘"라고 말하고 결과물을 받는 방식입니다. 간단한 것은 되지만, 복잡한 프로젝트에서는 한계가 있습니다.

- "로그인 기능 추가해줘" → 파일 20개 생성 → 어디에 뭐가 있는지 모름
- "버튼 색깔 바꿔줘" → 다른 기능이 망가짐 → 되던 게 안 됨
- 대화 100턴 넘어감 → AI가 앞에서 한 걸 까먹음 → 같은 실수 반복
- 3일 작업한 결과물 → 꼬여서 수습 불가 → 결국 처음부터 다시...

**SSAL Works는 체계적인 개발 방법론으로 이 문제를 해결합니다.**

SSAL Works는 Cursor, Bolt 같은 AI 코딩 도구가 아닙니다. AI 코딩 도구(Claude Code)를 활용한 개발 관리 플랫폼입니다.

## 2. SSAL Works의 정체성

### 2-1. 정의

SSAL Works는 **Claude Code를 활용한 3차원 SAL Grid 기반 풀스택 웹사이트 개발 실행 및 관리 플랫폼**입니다.

이 정의를 하나씩 풀어서 설명하겠습니다.

#### SSAL Works의 AI 개발 도구란?

SSAL Works는 프로젝트 개발을 위해 다양한 AI 개발 도구를 전략적으로 활용합니다. 주력 AI는 Claude Code이며, 보조적으로 Gemini CLI (보조 AI)를 활용하여 프로젝트의 효율성과 유연성을 극대화합니다.

**1. Claude Code (주력 AI):**
Claude Code는 Anthropic이 개발한 AI 도구입니다. 터미널에서 실행되며, 사용자의 지시를 받아 코드를 작성하고, 파일을 만들고, 프로젝트를 관리합니다.

**Claude Code의 역할:**

- **코드 작성**: HTML, CSS, JavaScript, TypeScript, Python 등
- **파일 생성 및 수정**: 프로젝트 파일 자동 생성
- **Git 버전 관리**: 커밋, 푸시, 브랜치 관리
- **터미널 명령 실행**: npm, git, 배포 명령 등
- **프로젝트 구조 관리**: 폴더 생성, 파일 정리
- **코드 리뷰**: 작성한 코드 검토
- **버그 수정**: 에러 분석 및 해결
- **테스트 실행**: 자동화 테스트 수행

**2. Gemini CLI (보조 AI):**
Google에서 개발한 AI 모델로, SSAL Works에서는 비용 효율적인 분석, 코드 리뷰, 설계 검토, 다국어 처리, 그리고 CLI 환경에서의 사용자 인터랙션 및 지시 이행 등 보조적인 역할을 수행합니다. Claude Code의 역량을 보완하고 개발 프로세스의 다양한 측면을 지원합니다.

**Claude Code vs 일반 Claude:**

- **Claude (웹/앱)**: 브라우저/앱에서 실행, 파일 접근 불가, 코드 실행 불가, Git/배포 작업 불가, 대화/질문/문서 작성 용도
- **Claude Code (터미널)**: 터미널(CLI)에서 실행, 로컬 파일 직접 접근 가능, 코드 직접 실행 가능, Git/배포 작업 직접 수행, 실제 개발 작업 용도

SSAL Works는 이 Claude Code를 핵심 도구로 사용하여, 비개발자도 풀스택 웹사이트를 개발할 수 있게 합니다.

#### 3차원 SAL Grid란?

SAL Grid는 SSAL Works가 개발한 프로젝트 관리 방법론입니다.

**SAL = Stage(단계) + Area(영역) + Level(레벨)**

```
     Area (Y)
        ↑
        │ ┌─┬─┬─┐
        │ ├─┼─┼─┤
        │ │T│ │ │←Level
        │ ├─┼─┼─┤
        │ └─┴─┴─┘
        └───→ Stage (X)
```

**Stage (X축 - 개발 단계):**

- **S1 (Development Setup, 개발 준비)**: 환경설정, DB 스키마, Auth 설정
- **S2 (Auth & Registration, 개발 1차)**: OAuth, 이메일, 회원가입
- **S3 (AI Integration, 개발 2차)**: AI 연동, AI Q&A
- **S4 (Payment & Admin, 개발 3차)**: 결제, 관리자, 크레딧
- **S5 (Development Stabilization, 개발 마무리)**: 배포, QA, 안정화

**Area (Y축 - 개발 영역):**

- **M (Documentation, 문서화)**: 기획서, 명세서 등
- **U (Design, UI/UX 디자인)**: 화면 설계, 디자인
- **F (Frontend, 프론트엔드)**: 사용자가 보는 화면
- **BI (Backend Infrastructure, 백엔드 기반)**: 서버 설정, 라이브러리
- **BA (Backend APIs, 백엔드 API)**: API 엔드포인트
- **D (Database, 데이터베이스)**: DB 스키마, 쿼리
- **S (Security, 보안/인증)**: 로그인, 권한 관리
- **T (Testing, 테스트)**: 단위/통합 테스트
- **O (DevOps, 운영/배포)**: 배포, CI/CD
- **E (External, 외부 연동)**: 외부 서비스 연동
- **C (Content System, 콘텐츠 시스템)**: 콘텐츠 관리

**Level (Z축 - 작업 순서):**

Level은 각 Task의 Stage별, Area별 순서를 나타냅니다.

예: S2F1, S2F2, S2F3 → Stage 2, Frontend Area의 1번, 2번, 3번 Task

#### SAL ID의 구조적 장점

SAL ID는 웹사이트 개발 프로젝트의 Stage(단계), Area(영역), Level(계층) 3차원 정보와 Variant(병렬분기)를 인코딩하여 절차적 순서, 의존성, 인접성, 병렬성 정보를 구조적으로 내재화합니다.

**SAL ID가 인코딩하는 정보:**

- **절차적 순서**: S1 → S2 → S3 순서로 진행
- **의존성**: 선행 Task와 후행 Task 관계
- **인접성**: 같은 Area 내 Task들의 연관성
- **병렬성**: 동일 Stage 내 독립 Task의 동시 실행 가능 여부

**결과적 효과:**

- ID만으로 작업의 위치와 맥락을 쉽고 빠르게 파악
- Task 지시서와 검증 지시서 체계적 생성
- 다수의 복잡한 작업을 최적화된 방식으로 진행
- 관리·통제·검증이 자동으로 이루어짐
- 프로젝트의 효율성과 효과성 극대화, 결과물 품질 향상

#### Task의 22개 속성

SAL Grid에서 각 Task는 22개의 속성으로 관리됩니다.

- **#1 task_id**: Task 고유 ID (예: S2F1)
- **#2 task_name**: Task 이름
- **#3 stage**: Stage 번호 (1~5)
- **#4 area**: Area 코드 (M, U, F, BI, BA, D, S, T, O, E, C)
- **#5 level**: Level 번호
- **#6 task_status**: 작업 상태 (Pending → In Progress → Executed → Completed)
- **#7 task_progress**: 진행률 (0~100)
- **#8 dependencies**: 선행 Task
- **#9 task_instruction**: Task 수행 지침
- **#10 task_agent**: Task 수행 Agent
- **#11 generated_files**: 생성된 파일
- **#12 duration**: 소요 시간
- **#13 build_result**: 빌드 결과
- **#14 verification_instruction**: 검증 지침
- **#15 verification_agent**: 검증 Agent
- **#16 test_result**: 테스트 결과
- **#17 build_verification**: 빌드 검증
- **#18 integration_verification**: 통합 검증
- **#19 blockers**: 차단 요소
- **#20 comprehensive_verification**: 종합 검증
- **#21 ai_verification_note**: AI 검증 의견
- **#22 stage_gate_status**: Stage Gate 상태

#### 왜 SAL Grid가 필요한가?

**체계 없는 코딩은 시행착오가 많습니다.**

AI가 아무리 뛰어나도 관리 시스템 없이 방치하면:
- 여기저기에 폴더와 파일을 만듭니다
- 일관성 없이 코드를 작성합니다
- 수정 작업이 매우 어려워집니다
- 어떤 파일이 어디에 있는지 찾기 힘듭니다

**AI도 관리 시스템이 필요합니다.**

SAL Grid는 모든 개발 작업(Task)을 3차원 좌표계 위에 배치하여:
- 체계적으로 관리합니다
- 모든 작업을 추적합니다 (Audit Trail)
- 작업자와 검증자를 분리합니다
- 단계별 품질을 보장합니다

#### 풀스택 웹사이트란?

프론트엔드(사용자가 보는 화면)와 백엔드(서버, 데이터베이스, API)가 모두 갖춰진 완전한 웹사이트입니다.

**풀스택 웹사이트의 구성:**

```
┌──────────────┐
│  Frontend    │
│  (화면)      │
└──────┬───────┘
       ↕
┌──────┴───────┐
│  Backend     │
│  (서버/API)  │
└──────┬───────┘
       ↕
┌──────┴───────┐
│  Database    │
│  (DB)        │
└──────────────┘
```

단순한 랜딩 페이지가 아니라, 회원가입, 로그인, 결제, 데이터 저장, 관리자 기능 등 실제 비즈니스를 운영할 수 있는 웹사이트입니다.

### 2-2. SSAL이라는 이름의 의미

SSAL이라는 이름은 두 가지 의미가 결합되어 있습니다.

#### 첫째, S + SAL

S는 개발자 Sunny의 성씨 첫 글자이고, SAL은 3차원 좌표계(Stage, Area, Level)의 약자입니다. 따라서 SSAL = Sunny + Stage + Area + Level입니다.

#### 둘째, 쌀

SSAL의 한글 발음은 "쌀"입니다. 쌀이 한국인의 핵심 먹을거리이듯, 웹사이트는 사업의 핵심 먹을거리입니다. 쌀이 있어야 밥을 먹듯, 웹사이트가 있어야 수익을 냅니다. SSAL Works에서 만드는 웹사이트는 사용자에게 돈을 벌어다주는 핵심 먹을거리가 됩니다.

### 2-3. 황금쌀 방앗간

SSAL Works를 비유적으로 표현하면 **"황금쌀 방앗간"**입니다.

```
  농부 (사람)
      │
      ↓ 벼 (아이디어)
 ┌──────────┐
 │ 방앗간   │
 │SSAL Works│
 └────┬─────┘
      ↓ 쌀 (웹사이트)
  황금쌀 (수익)
```

농부(빌더)가 벼(창의적 아이디어)를 가져오면, 방앗간(SSAL Works)에서 방아(AI)가 찧어서 쌀(풀스택 웹사이트)을 만들어냅니다. 그리고 그 쌀은 황금쌀, 즉 돈을 버는 웹사이트가 됩니다.

방앗간에서 농부가 직접 방아를 찧지 않아도 쌀이 나오듯이, SSAL Works에서 사용자가 직접 코딩을 하지 않아도 웹사이트가 만들어집니다.

### 2-4. 인간과 AI의 역할 분담

SSAL Works에서 인간과 AI는 각자의 역할이 명확하게 분담되어 있습니다.

#### AI(Claude Code)가 담당하는 것

- **코딩**: HTML, CSS, JavaScript, TypeScript 등 작성
- **문법과 문서 형식**: 올바른 문법, 표준 형식 준수
- **도구 사용**: Git, npm, Vercel CLI 등 도구 활용
- **반복적인 작업**: 파일 생성, 코드 수정 반복
- **기술적인 구현**: 알고리즘, 로직 구현
- **에러 해결**: 버그 분석 및 수정
- **테스트**: 자동화 테스트 수행

#### 인간(빌더)이 담당하는 것

- **기획**: 무엇을 만들 것인지 결정
- **의사결정**: 어떤 방향으로 갈 것인지 결정
- **비즈니스 모델**: 어떻게 수익을 낼 것인지 설계
- **Order Sheet 작성**: AI에게 지시할 내용 정리
- **최종 승인**: 결과물 검토 및 승인
- **외부 서비스 설정**: API 키 발급, OAuth 설정 등

## 3. SSAL Works가 제공하는 것

### 3-1. 핵심 방법론 및 체계

SSAL Works는 4가지 핵심 방법론과 체계를 기반으로 합니다.

**3-1-1. SAL Grid**

3차원 좌표계(Stage, Area, Level)로 프로젝트 전체를 관리하는 방법론입니다.

**핵심 특징:**
- 각 Task를 22개 속성으로 체계적으로 관리
- Task Agent(작업자)와 Verification Agent(검증자) 분리
- 모든 작업의 Audit Trail 보장으로 추적 가능
- 단계별 Stage Gate 승인 절차
- 철저한 문서화 및 기록

**Task 상태 전이:**

```
Pending
   ↓
In Progress
   ↓
Executed (작업 완료)
   ↓
Completed (검증 후)
```

**검증 상태 전이:**

```
Not Verified
   ↓
In Review
   ↓
Verified
(또는 Needs Fix)
```

**3-1-2. Briefing(안내문)**

각 개발 단계마다 무엇을 해야 하는지 상세히 안내하는 문서입니다.

**"뭘 해야 할지 몰라서 멍하니 있는 시간"**을 없앱니다. 회원가입 후 환영 메시지, 프로젝트 등록 후 플랫폼 사용법, P0~P3 기획 단계별 안내, S0~S5 개발 단계별 안내, 완료된 프로젝트 수정 등 상황별 안내문이 준비되어 있습니다.

**3-1-3. Order Sheet**

AI에게 체계적으로 작업을 지시하는 표준화된 양식입니다.

AI에게 "이거 해줘"라고만 하면 엉터리가 됩니다. Order Sheet는 AI가 지켜야 할 규칙을 명확하게 정의하고, 작업 내용을 구체적으로 기술하며, 기대 결과물을 명시합니다.

**Order Sheet 구성:**
- **Part A**: 모든 프로젝트에 공통으로 적용되는 표준 규칙 (7대 Rules + 3대 Methods + 12대 Compliance + 5대 절대 규칙)
- **Part B**: 프로젝트별로 커스터마이징하는 부분 (프로젝트 정보, 작업 지시, 참고 파일, 기대 결과물)

**3-1-4. 데이터 로컬 저장**

모든 프로젝트 자료가 서비스 회사 서버가 아닌 빌더의 컴퓨터에 저장됩니다.

| 걱정 | SSAL Works 답변 |
|------|----------------|
| "서비스 망하면 어떡해요?" | 파일은 내 컴퓨터에 있어서 괜찮습니다 |
| "구독 끊으면 못 보나요?" | 아니요, 파일은 계속 내 것입니다 |
| "데이터 유출 걱정돼요" | 서비스 회사 서버에 안 올라가니까 안전합니다 |
| "나중에 옮기기 어렵나요?" | 폴더 복사만 하면 됩니다 |

**유일하게 수집하는 정보:** 진행률만 수집합니다. 파일 내용, 코드 내용, 기획 문서 내용은 절대 수집하지 않습니다.

Claude Code(AI)와 함께 웹 프로젝트를 처음부터 끝까지 수행할 수 있도록 설계된 **통합 개발 환경 패키지**입니다. **"다운로드 → 압축 해제 → Claude Code 실행"** 3단계로 프로젝트 개발 환경이 완성됩니다.

- 프로젝트 디렉토리 구조 (P0~S5 총 10단계)

**예상 절약 시간: 4~6주**

**3-2-4. 10단계 진행 프로세스 진척도 확인**

P0~P3(기획 4단계) + S0~S5(개발 6단계)의 10단계로 프로젝트 진도를 관리합니다. 각 단계별 진척률이 자동으로 계산되어 대시보드에 표시됩니다.

- Task별 작업 상태 (Pending → In Progress → Executed → Completed)
- 검증 상태 (Not Verified → In Review → Verified)
- 작업 지시서 및 검증 지시서
- 생성된 파일 목록
- 차단 요소 및 해결 현황

### 5-2. 월 이용료

## 6. 용어 정리

**주요 용어**

- **SSAL Works**: Claude Code 활용 3차원 SAL Grid 기반 풀스택 웹사이트 개발 플랫폼
- **SAL Grid**: Stage, Area, Level 3차원 좌표계 프로젝트 관리 방법론
- **회원**: SSAL Works에 무료 가입한 사용자
- **빌더 계정 개설자**: 개설비를 납부하고 빌더 계정을 활성화한 사용자
- **PO (Project Owner)**: 빌더 계정을 개설하여 프로젝트를 진행하는 사람. 빌더 계정 개설자와 동일
- **프로젝트**: 빌더가 등록한 웹사이트 개발 프로젝트
- **개발**: AI와 협업하여 웹사이트를 만드는 과정
- **Task**: SAL Grid의 개별 작업 단위
- **Order Sheet**: AI에게 작업을 지시하는 표준화된 양식
- **Briefing**: 각 단계별 안내문
- **Dev Package**: AI 협업 웹 개발 올인원 패키지
- **Stage Gate**: 각 Stage 완료 시 수행하는 품질 검증 시스템
- **Sunny**: SSAL Works 개발자 겸 창업자
